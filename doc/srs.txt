1.     Introduction
1.1     Purpose
 This Software Requirements Specification (SRS) document outlines the functional and non-functional requirements for the Group Finance Tracking Application, version 1.0. The purpose of this application is to provide a digital platform for groups of individuals — such as roommates, travel groups, teams, or families — to collaboratively manage shared expenses, budgets, and financial responsibilities.


This SRS covers the full scope of the application, including the mobile platform. It defines the requirements for key features such as group creation, shared expense tracking, debt settlements, notifications, and reporting.

 

The document is intended for use by the development team, project managers, testers, UI/UX designers, and stakeholders. It ensures that all team members and stakeholders have a shared understanding of the system's capabilities and constraints.

 

If this SRS is revised or expanded in the future, it may include integration with third-party payment services or banking APIs, which are out of scope for the current version.

1.2     Document Conventions
This document uses numbered headings for organization and bold text to highlight key terms. Requirements are stated clearly using “shall” for mandatory and “should” or “may” for optional features. Each requirement is individually listed with its own priority, if applicable. Acronyms are defined on first use and included in the Glossary.

1.3     Project Scope
The Group Finance Tracker is a personal finance application designed to simplify shared expense management. It reads SMS messages to automatically detect bills and allows users to assign participants for each expense. Users can later generate individual bills showing each person’s share. The app aims to reduce manual tracking, improve transparency, and streamline cost-sharing among friends, families, or roommates. This SRS defines the requirements for the initial release.

2.     Overall Description
2.1     Product Perspective
This product is a new, self-contained mobile application designed to track personal and group expenses by reading transaction-related SMS messages on Android devices and providing a manual entry option on iOS devices.

It operates entirely offline, storing all data locally using Drift for structured storage and enabling expense management without requiring internet connectivity.

The application integrates the following core components:

·         SMS Listener (Android only) – Runs in the background to detect transaction-related messages and trigger expense notifications.

·         Expense Manager – Allows manual entry, editing, and deletion of expenses, including source of payment and split options.

·         Pending Expense Handler – Stores unconfirmed SMS-based expenses for later review.

·         Bill Sharing Module – Generates PDF or text bills for entire groups or individual members and allows sharing via other apps.

·         Dashboard & Analytics – Displays monthly spending totals, source-based spending distribution, and recent activity.

2.2     Product Features
This application is a new, self-contained product designed to help individuals and groups manage shared expenses efficiently. It is not part of an existing system or product family.

The app leverages SMS parsing to automatically identify transaction or bill notifications, reducing the need for manual entry. It allows users to tag participants for each bill, and later generate individual payment summaries to facilitate easy settlements.

The product is intended to work independently on mobile devices (Android initially), with potential future integration with payment platforms, contact lists, or cloud backup services.

Key components of the system include:

·         SMS Reader module

·         Expense Parser and Categorizer

·         Group & Member Management

·         Bill Split Engine

·         Report Generator

Each of these components works together to provide a seamless user experience for tracking and splitting group expenses.

2.5     Design and Implementation Constraints
2.5.1 Platform Restrictions:

·         iOS does not permit automatic access to SMS inbox; bill detection must be manual or user-assisted on iOS.

·         Android requires runtime permissions to access SMS, which must be handled securely and transparently.

2.5.2 Local-Only Architecture:

·         The app must function completely offline with no reliance on cloud storage or external servers.

·         All user data must be stored and processed locally on the device.

2.5.3 Storage Technology:

·         Drift (Moor) is the selected local database framework due to its SQL-style querying and Flutter compatibility.

·         Storage must be optimized for performance on low-end devices.

2.5.4 Security and Privacy:

·         User financial data must never leave the device.

·         SMS data must be accessed only with explicit user permission.

·         All data access must comply with OS-level permission models and privacy standards.

·         No telemetry or background network activity is permitted unless explicitly user-initiated (e.g., exporting PDF).

2.5.5 Cross-Platform Development:

·         The app must be fully implemented in Flutter to support both Android and iOS from a single codebase.

·         All third-party packages must support both platforms natively.

2.5.6 Device Limitations:

·         Must run efficiently on low-to-mid-range smartphones (min. 2GB RAM, limited storage).

·         Background SMS detection must be optimized for battery and performance.

2.5.7    Background Operations:

·         On Android, the app must be able to process SMS messages in the background and:

o   Detect transaction patterns

o   Generate contextual notifications

o   Temporarily store extracted bill data for later confirmation

·         These operations must be power-efficient and not abuse background services.

2.5.7 Design Standards:

·         UI must follow Material Design for Android and Cupertino guidelines where appropriate on iOS.

·         Bill parsing accuracy and clarity in notifications are critical to user trust.

·         Code must follow clean architecture and Dart best practices for maintainability.

 

2.6     User Documentation
 The following user documentation components will be provided to assist users in understanding and using the Group Finance Tracker effectively:

1.    In-App Onboarding Guide:

o    A simple first-time walkthrough introducing key features such as SMS-based expense detection, bill splitting, and generating individual summaries.

2.    Help Section:

o    Accessible from the app’s settings menu.

o    Contains FAQs and short usage instructions (e.g., how to tag participants, manage expenses, or edit entries).

3.    Minimal In-App Hints:

o    Lightweight labels or info icons near interactive elements (e.g., the split-evenly toggle slider) to guide users without overwhelming them.

3.     System Features
3.1     SMS-Based Bill Detection (Android Only)
3.1.1      Description and Priority

The application listens to incoming SMS messages and automatically detects transaction-related texts (e.g., debits, purchases). When a valid transaction is detected, a local notification is triggered.

3.1.2      Stimulus/Response Sequences

 

Stimulus

System Response

1. An SMS containing a transaction message (e.g., debit, payment) is received

App's background SMS listener is triggered and parses the message content.

2. Message is identified as a bill or expense (based on keywords, patterns)

App extracts bill amount, merchant info, date, and time from the SMS.

3. Parsed data is valid and matches known transaction patterns

App stores the bill temporarily in local memory as a draft expense.

4. A draft bill is successfully created

App shows a notification to the user with a summary and an action: “Tap to confirm and add this expense.”

5. User taps the notification

App opens the “Add Expense” screen pre-filled with parsed data (amount, date).

6. User ignores the notification

Draft bill remains stored locally and can be accessed later from a “Pending Bills” section.

7. User denies SMS permission or disables background processing

App cannot detect or notify on new bills; user must manually add expenses.

 

3.1.3      Functional Requirements

·         The system shall monitor SMS messages in the background.

·         The system shall parse and extract key details such as amount, merchant name, and timestamp.

·         The system shall store a draft bill entry for user review.

·         The system shall notify the user with an actionable prompt to add or ignore the detected expense

·         The system shall allow users to later review SMS-detected bills stored as drafts and discard them if identified as false positives.

3.6     Notifications
3.6.1      Description and Priority

The application includes a real-time notification feature that alerts users when a transaction-related SMS is detected and interpreted as a potential bill. These notifications are designed to help users quickly act on new expenses without manually opening the app. Each notification includes actionable options such as confirming or discarding the expense.
The goal is to improve user engagement and ensure bills are captured promptly, while minimizing the clutter of false positives

3.6.2      Stimulus/Response Sequences

Stimulus

System Response

A draft bill is created (via SMS detection)

App shows a notification summarizing the transaction, with options: “Add” and “Discard”

User taps “Add”

App opens “Add Expense” screen with pre-filled data

User taps “Discard”

App discards the transaction as a false positive and does not create an expense backlog

User ignores or dismisses the notification

No action taken; draft bill remains in “Pending Bills” section for later review

 

 

3.6.3      Functional Requirements

·         The system shall detect transaction-related SMS messages and generate notifications when a valid bill is identified.

·         The system shall display a notification with a summary of the transaction and provide two action buttons: “Add” and “Discard”.

·         When the user taps “Add”, the app shall open the “Add Expense” screen with pre-filled data from the SMS.

·         When the user taps “Discard”, the app shall treat the SMS as a false positive and not save the transaction.

·         If the notification is dismissed or ignored, the app shall store the extracted bill as a pending draft, allowing the user to review and take action later.

3.7     Pending Bills Management
3.7.1      Description and Priority

The Pending Bills section stores all bills that were detected via SMS but not confirmed by the user. This feature provides users with the ability to manage unconfirmed expenses, helping reduce false data entries and improving control over bill tracking

3.7.2      Stimulus/Response Sequences

Stimulus

System Response

User opens the "Pending Bills" section

App displays a list of unconfirmed bills with basic details

User taps on a bill

App shows full parsed data and options: Add or Discard

User taps “Add”

App opens the Add Expense screen with pre-filled values

User taps “Discard”

App deletes the bill from memory and removes it from the pending list

 

 

3.7.3      Functional Requirements

·         The system shall display all unconfirmed bills in a "Pending Bills" section.

·         The system shall allow users to review each pending bill and take one of the following actions:

o    Confirm and Add: Opens the expense entry form pre-filled with SMS data.

o    Discard: Permanently deletes the bill from storage, treating it as a false positive.

·         The system shall update the local database to reflect the user's action (add or discard).

·         The system shall not show discarded bills again or retain them in memory.

·         The system shall display a visual breakdown of spending by source for the current month.

·         The user shall be able to switch between the “Total Monthly Spending” view and “Spending by Source” view on the dashboard.

5.     Other Nonfunctional Requirements
5.1     Performance Requirements
·         The app should process and store incoming SMS transaction data in under 1 second after receipt.

·         Bill generation (PDF or TXT) must be completed within 3 seconds for up to 500 expense entries.

·         Background SMS monitoring should consume less than 2% of daily battery usage under normal operation.

·         Home dashboard loading time should be under 1.5 seconds.

 

5.2     Safety Requirements
·         The app will not modify or delete SMS messages; it will only read and parse them.

·         In case of unexpected app termination, partially entered expenses will be saved in local storage for recovery upon restart.

·         Bill sharing will only proceed after explicit user action to prevent accidental sharing of sensitive data

 

5.3     Security Requirements
·         All local data storage will be encrypted at rest using AES-256.

·         All generated PDF/TXT files will be stored in a temporary share folder and deleted automatically after 24 hours if not manually saved by the user.

·         The app will request only the permissions strictly necessary for its operation (e.g., SMS read, notifications, storage access for sharing).

·         User activity, such as discarding false positives, will remain private and not be sent to any server.

·         Usability:

·         The app will provide in-app tips for first-time users.

·         Reliability:

·         Background SMS detection should have at least 95% accuracy in identifying relevant transaction messages.

·         Maintainability:

·         Code will be modular to allow future integration of cloud backup if needed.

·         The database layer will use Drift for maintainable local storage management.

·         Portability:

·         The app will be fully functional on Android (primary target).

·         iOS support will be provided with feature parity except for SMS background reading (due to platform restrictions).
